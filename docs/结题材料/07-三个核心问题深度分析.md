# 三个核心问题深度分析

本文档围绕 STrans 项目开发和迭代中面临的三个核心问题，分别阐述问题背景、影响范围、已采取的技术方案、迭代过程中的改进思路和当前效果评估。

## 问题一：沙盘环境下小目标车辆检测与车牌识别的鲁棒性

### 1.1 问题描述

智慧交通沙盘中的模型车辆尺寸通常在图像中仅占 30×20 至 80×60 像素区域，且存在以下挑战：

- **目标尺寸小**：沙盘比例约 1:87，标准小轿车在 1080p 画面中仅占数十像素；
- **遮挡频繁**：树木、建筑、桥梁和前后车辆相互遮挡，导致目标不完整；
- **光照变化大**：沙盘灯光、窗边自然光和阴影导致车辆外观特征不稳定；
- **车牌尺寸极小**：沙盘车牌仅占几个像素，OCR 极易失败；
- **多车排队场景**：路口排队时车辆紧密相邻，易产生重叠框和漏检。

### 1.2 技术方案

#### 1.2.1 低阈值保召回 + 多层后置过滤

**问题**：直接使用 YOLO 默认置信度阈值（0.5），沙盘小车的召回率严重不足；降低阈值至 0.25–0.35 又会引入大量树木、灯杆和道路设施的误检。

**方案**：采用"低阈值保召回 + 多层后置规则过滤"策略：

1. 检测阈值设为 0.35（默认），道路异常模式进一步降至 0.25；
2. **道路 ROI 过滤**：检测框必须落入预定义的道路感兴趣区域（见 `road_anomaly._road_roi_mask`），排除路边建筑和树木；
3. **面积过滤**：框面积过小的噪声被直接过滤（见 `local_model._filter_detections`）；
4. **外观可信度判断**（`local_model._appearance_conf`）：利用边缘密度、对比度和熵三个特征判断检测框内是否是真实车辆：
   - 边缘密度过低 → 可能是路面纹理、阴影
   - 对比度过低 → 可能是均匀的树木或墙面区域
   - 熵过低 → 可能是单色反光或灯杆

#### 1.2.2 ByteTrack 轨迹管理与时序稳定

**问题**：单帧检测结果不稳定——同一辆车可能因遮挡或角度变化在不同帧中置信度波动，甚至短暂消失。

**方案**：引入 ByteTrack 多目标跟踪（`local_model._track_model`），并自行补充时序稳定机制：

1. **视觉 ID 恢复**（`local_model._recover_visual_ids`）：ByteTrack 丢失 ID 时，通过框位置 IoU 匹配恢复短时视觉身份；
2. **轨迹框平滑**（`local_model._smooth_track_boxes`）：使用 EMA（指数移动平均）平滑同一轨迹的框位置；
3. **短时预测补偿**（`local_model._predict_missing_tracks`）：对已证明可信的轨迹，在 1–3 帧漏检时用上一帧位置预测，并随时间衰减置信度；
4. **车辆数中值稳定**（`local_model._stable_vehicle_count`）：最近 5 次观测取中值，避免单帧波动；
5. **重叠框抑制**：合并 IoU ≥ 0.5 的重叠车辆框和车牌反推框。

#### 1.2.3 车牌识别的多层策略

**问题**：沙盘车牌极小，单一 OCR 策略无法兼顾全部场景。

**方案**：构建"车辆裁剪 OCR → 轨迹记忆 → 全帧 OCR 补充"三层策略（`local_model.detect_plates` 链路）：

1. **车辆裁剪 OCR**：对每个已检测车辆区域调用 HyperLPR3，是主识别路径；
2. **OCR 预算与轮转**（`_ocr_budget`）：实时流每轮限制 OCR 数量（默认 3 个），按轨迹置信度和上次 OCR 时间轮转优先级，让排队后车也有机会被识别；
3. **轨迹车牌记忆**（`_stable_track_plate`）：同一轨迹的 OCR 文本进入缓冲区，经过文本修正（`fix_plate_text`）、多帧一致性确认（2 次以上相同结果）和过期清理后形成稳定车牌；
4. **全帧 OCR 补充**（`_full_frame_ocr_plates`）：存在未识别车辆时周期性执行全帧 OCR；若结果能归属已有车辆则补齐字段，否则从车牌位置反推车辆候选框；
5. **白名单决策稳定**（`_stable_decision`）：稳定车牌查询 SQLite 白名单，短时缓存（5 秒）防止识别波动导致通行决策跳变。

#### 1.2.4 远近视角自适应

**问题**：部分摄像头（如 live3）为远景视角，车辆在画面中极小。

**方案**：自动判断远景视角（画面宽高比和内容特征），裁剪道路重点区域并将推理尺寸至少提高到 960（`local_model._distant_view_crop`），使 YOLO 的输入中目标相对尺寸增大。

### 1.3 迭代过程

见 [03-算法迭代与提交历史.md](03-算法迭代与提交历史.md) 第 3.1–3.2 节。核心演进路径：

```
单帧高阈值检测 → 低阈值+道路过滤 → +ByteTrack跟踪
  → +视觉ID恢复+框平滑 → +外观可信度
  → +重叠抑制+中值统计 → +裁剪OCR+轨迹记忆
  → +全帧OCR+车牌反推 → +白名单决策缓存
  → +远近自适应+OCR预算轮转
```

### 1.4 当前效果与局限

- **优势**：在沙盘各类视角下可稳定检测大部分可见车辆，遮挡不严重时跟踪连续，车牌在白名单内的车辆通行决策正确；
- **局限**：严重遮挡（>70%）和极远景车辆仍可能漏检；非参考摄像头的速度值为估计值；OCR 对沙盘小牌存在固有精度上限。

## 问题二：道路异常检测的误报控制

### 2.1 问题描述

道路异常检测需要在交通场景中找出"不应该出现在路面上的物体"，但以下因素极易触发误报：

- **车道标线和箭头**：白色标线在背景差分中表现为高亮变化区域；
- **光照和阴影**：沙盘灯光变化、窗边自然光移动导致差分出现大面积变化；
- **路面反光**：沙盘路面为光滑塑料材质，灯光下产生镜面反光；
- **车辆移动后留下的"空位"**：合法车辆开走后，背景差分显示原位置有"变化"；
- **相机轻微抖动**：沙盘底座震动导致画面整体微移，差分满屏触发。

### 2.2 技术方案

#### 2.2.1 全局相机运动抑制

**问题**：手机/移动摄像头或沙盘震动导致的画面整体移动。

**方案**（`road_anomaly._frame_difference_candidates`）：

1. **稀疏光流**：使用 Lucas-Kanade 光流法跟踪角点，计算全局运动向量；
2. **中位位移判断**：光流中位位移超过阈值（2.5 像素）时暂停异常检测；
3. **变化比例检测**：若道路 ROI 内变化像素比例超过 8.5%，视为相机运动或光照突变，不输出异物。

#### 2.2.2 合法道路使用者解释

**问题**：车辆和行人是正常道路使用者，不应被误判为异物；车辆离开后的"空位"也不应触发告警。

**方案**：

1. **YOLO/ByteTrack 结果作为解释器**：将当前帧的车辆和行人检测框从异常候选区域中扣除；
2. **近期合法框记忆**（`CameraState.recent_legal_boxes`）：保留最近离开的车辆/行人框，在其原位置短暂抑制异常检测（防止车辆刚开走时触发"异物"）；
3. **重叠判断**：候选区域与合法框 IoU ≥ 0.25 则被剔除。

#### 2.2.3 多线索外观过滤

**问题**：车道线、反光、阴影等在差分中表现为局部变化区域，需要通过视觉特征进一步排除。

**方案**：在背景差分和形态学处理后，对每个候选区域进行外观验证：

| 过滤维度 | 方法 | 排除的目标 |
|---|---|---|
| 颜色 | Lab 色度距离、HSV 暖色通道 | 白色标线、灰色阴影（与路面颜色接近） |
| 纹理 | 局部熵、对比度 | 平滑反光面、均匀阴影 |
| 形状 | 轮廓填充率、圆度、凸包实心度 | 细长车道线、圆形灯光光斑 |
| 亮度 | 相对于局部邻域的亮度比值 | 纯白反光点 |
| 专项 | 车道线和箭头方向梯度特征 | 方向性标线图案 |

#### 2.2.4 多帧一致性确认

**问题**：单帧的差分结果可能是压缩噪声或瞬间反光。

**方案**：
- 实时流候选需连续 3 帧以 IoU ≥ 0.30 匹配才确认为道路异物；
- 静态图片候选连续 2 帧 IoU ≥ 0.35 后确认；
- 无异常时以低权重（0.015）更新背景模型，防止异常物体逐渐被"吸收"到背景中。

#### 2.2.5 静态图片独立分支

**问题**：静态图片（JPG/PNG）没有时间差信息，帧差法完全无效。

**方案**（`road_anomaly._static_scene_candidates`）：
- 亮度通道提取高亮实体；
- Lab 色度距离发现与主路面颜色不同的物体；
- HSV 暖色通道针对沙盘验收中常见的橙黄小物体（乒乓球/异物）；
- 通过轮廓填充率、圆度、凸包实心度、对比度、熵和白色占比过滤车道线、箭头和反光；
- 已确认车辆框和车牌区域作为"可解释目标"从掩膜中扣除。

### 2.3 迭代过程

见 [03-算法迭代与提交历史.md](03-算法迭代与提交历史.md) 第 3.4 节。核心演进路径：

```
简单ROI帧差 → +形态学去噪
  → +光流全局运动抑制 → +变化比例光照检测
  → +合法目标(YOLO/ByteTrack)解释
  → +颜色/纹理/形状多线索过滤
  → +车道线箭头专项过滤
  → +多帧IoU确认 → +静态图片独立分支
  → +近期合法框记忆 → +可选道路破损模型
```

核心提交 `f87aa0c`「优化道路异物检测稳定性」是这一问题的最新优化节点。

### 2.4 当前效果与局限

- **优势**：在固定沙盘摄像头、光照稳定场景下，真实异物的检出率高、误报率可控；多线索融合使系统具备可解释性——每一层过滤都有明确的物理意义；
- **局限**：光照剧烈变化（开关灯）时仍可能误报；阴影与深色异物（如棕色小物体）在颜色过滤中可能被误排除；阈值参数依赖沙盘场景经验，迁移到新场景需要重新调参。

## 问题三：多源异构视频接入与完整业务闭环

### 3.1 问题描述

STrans 需要同时支持多种视频源类型，并形成"采集—识别—分析—处置—归档"的完整业务闭环：

- **视频源异构**：RTSP 流、MJPEG 流、手机 IP Webcam、本地图片、本地录像，协议和格式各不相同；
- **管理需求**：摄像头增删改查、连通性测试、批量启停、状态展示；
- **数据闭环**：识别结果不仅要展示，还需持久化、可查询、可导出，告警需能处置和追溯；
- **权限体系**：区分普通用户和管理员，操作需审计记录；
- **系统稳定性**：多路视频同时运行时资源受限，需要智能调度。

### 3.2 技术方案

#### 3.2.1 统一视频接入层

**方案**：`CameraHub` + `VideoStreamService` 双层架构。

```
CameraHub (管理层)
  ├── _sources: dict[camera_id, CameraSource]    # 摄像头配置
  ├── _streams: dict[camera_id, VideoStreamService] # 视频流实例
  ├── 配置持久化 (SQLite)
  ├── 集中启停和状态管理
  └── MJPEG 帧分发

VideoStreamService (流层)
  ├── OpenCV VideoCapture 统一解码
  ├── RTSP/MJPEG/图片/录像 自动识别
  ├── 断流恢复和重连
  └── 帧缓存和最新帧快照
```

关键设计：
- `VideoStreamService` 支持 `rtsp://`、`http://`、本地文件路径的自动识别；
- `CameraHub` 管理多路流，提供 `latest_jpeg()` 方法为算法模块和前端提供最新帧；
- 预置 4 路沙盘 RTSP 摄像头（SANDTABLE_CAMERAS），支持动态增删自定义源；
- 视频源地址在日志和审计中脱敏（`video_stream._redact_url`），防止凭据泄露。

#### 3.2.2 统一数据结构 `AnalysisResult`

**方案**：所有算法模块统一输出 `AnalysisResult`（见 `backend/app/schemas/dashboard.py`）：

```python
class AnalysisResult(BaseModel):
    frame_id, timestamp, camera_id, model_id
    inference_ms, source_width, source_height
    detections: list[DetectionBox]    # 检测框+车牌+白名单+速度
    traffic_stats: TrafficStats       # 车辆数/密度/拥堵
    events: list[TrafficEvent]        # 异常和业务事件
    raw: dict                         # 扩展元数据
    error: str | None                 # 统一错误通道
```

这一结构使车辆监控和道路异常两种任务模式的结果可以统一存储、统一展示、统一导出。

#### 3.2.3 数据闭环设计

**方案**：以 SQLite 为核心，构建"结果→历史→告警→证据→处置"链路。

| 环节 | 数据表/存储 | 核心操作 |
|---|---|---|
| 识别结果 | `analysis_records` | 每 5 秒自动保存，含检测框、统计、事件 JSON |
| 告警事件 | `alert_incidents` | 道路异物/行人/破损/白名单事件自动入库 |
| 处置流转 | `alert_incidents.status` | open → acknowledged → resolved，记录处理人和备注 |
| 证据包 | ZIP (原始帧+标注帧+JSON+SHA-256) | 按 event_id 关联，含完整性校验 |
| 历史查询 | API `/api/history` | 按摄像头筛选、CSV/JSON 导出 |
| 智能报告 | `intelligence_reports` | DeepSeek 根据实时数据生成归档报告 |

#### 3.2.4 权限与审计体系

**方案**：

- **RBAC 权限**：admin/user 两种角色，使用 FastAPI `Depends` 注入身份验证和权限检查；
- **图片验证码**：注册和登录需验证码，防自动化攻击；
- **密码安全**：SHA-256 + 随机盐哈希，管理员可重置用户密码；
- **审计日志**（`audit_logs`）：摄像头修改（含脱敏）、白名单变更、用户管理、告警处置、证据导出等操作均有审计记录。

#### 3.2.5 自适应模型调度

**方案**（见 `adaptive_scheduler.py`）：应对多路运行时资源波动。

| 策略 | 触发条件 | 动作 |
|---|---|---|
| quality | 静态图 或 GPU 余量充足 | 推理尺寸 ≥ 768–960，提高小目标精度 |
| anomaly | 道路异常任务 | 保留 768–960 分辨率，限制检测频率 |
| protect | 显存≥88% 或 内存≥92% 或 CPU≥92% | 切 fallback 模型，尺寸≤512，间隔≥350ms |
| realtime | GPU≥88% 或 推理≥150ms | 尺寸≤640，间隔≥180ms |
| balanced | 资源正常 | 保持原配置 |
| manual | 调度关闭 | 使用人工配置 |

滞回机制：非 protect 策略切换需 8 秒冷却，防止频繁抖动；资源数据缓存 1.5 秒。

### 3.3 迭代过程

见 [03-算法迭代与提交历史.md](03-算法迭代与提交历史.md)。核心演进路径：

```
视频闭环(接入+前端) → 核心感知集成
  → 业务闭环(历史+告警+账号+白名单)
  → 移动视角热力图 → 道路语义约束
  → 自适应调度 → 摄像头管理CRUD
  → 证据链+审计 → 录像保留与导出
```

### 3.4 当前效果与局限

- **优势**：已形成完整的"采集→识别→分析→处置→归档"闭环；支持管理员和普通用户两种角色；所有识别结果可回溯、告警可处置、证据可验证；
- **局限**：未接入物理闸机执行真实放行/拦截动作；智能报告依赖外部 DeepSeek API 可用性；长稳运行（30 分钟+）尚未充分验证。

## 总结：三个问题的整体解决思路

三个问题对应项目从"能演示"到"工程化"的三个层次：

```
问题一 (算法精度) → 车辆/车牌识别的鲁棒性
    策略：模型(召回) + 几何(ROI) + 时序(跟踪) + 外观(可信度)
          + 多层OCR + 白名单决策

问题二 (误报控制) → 道路异常检测的可信度
    策略：运动(光流) + 解释(YOLO) + 外观(颜色/纹理/形状)
          + 时序(多帧确认) + 场景(静态/实时分支)

问题三 (系统工程) → 多源接入与业务闭环
    策略：统一接入(CameraHub) + 统一数据(AnalysisResult)
          + 持久化(SQLite) + 权限(RBAC) + 调度(自适应)
```

三者并非独立解决——问题一和问题二共享 YOLO/ByteTrack 结果作为"合法目标解释"；问题三的数据闭环使问题一、二的检测结果可追溯、可验证；自适应调度则在资源受限时保障前两个问题的持续运行。
