# 各模块实现思路详解

本文档按系统分层逐一阐述各核心模块的设计思路、关键数据流和实现要点，作为系统设计说明书的详细补充。

## 1. 视频接入层

### 1.1 CameraHub（摄像头管理中心）

**文件**：`backend/app/services/camera_hub.py`
**职责**：多路摄像头全生命周期管理

**设计思路**：

```
┌─────────────────────────────────────────┐
│               CameraHub                  │
│  ┌───────────────────────────────────┐  │
│  │  _sources: dict[str, CameraSource]│  │  ← 摄像头配置（内存+SQLite）
│  │  _streams: dict[str, VideoStream] │  │  ← 活跃视频流实例
│  └───────────────────────────────────┘  │
│                                          │
│  配置持久化 ←→ camera_sources 表         │
│  流管理      ←→ VideoStreamService       │
│  帧分发      →  latest_jpeg(camera_id)   │
│  MJPEG 输出  →  mjpeg_frames(camera_id)  │
└─────────────────────────────────────────┘
```

**关键实现**：

- **预置源加载**：系统启动时从 `SANDTABLE_CAMERAS`（4 路沙盘 RTSP）初始化，再从 SQLite `camera_sources` 表加载自定义摄像头；
- **配置持久化**：增删改操作同步写入 SQLite，字段包括 `camera_id, name, type, stream_url, location, heatmap_mode`；
- **连接管理**：每个摄像头对应一个 `VideoStreamService` 实例，复用 `open_sqlite` 统一连接类保证资源释放；
- **预置源保护**：沙盘预置摄像头（type=sandtable）不可删除，仅允许修改部分字段；
- **热力图模式**：每个摄像头独立存储 `heatmap_mode`（auto/off/overlay/schematic），前端按模式渲染。

**新增源码文件**：`sqlite_utils.py` 提供 `ClosingSQLiteConnection`——解决 Windows 下 SQLite 连接未及时关闭导致临时数据库无法删除的问题（修复 BUG-001）。

### 1.2 VideoStreamService（视频流引擎）

**文件**：`backend/app/services/video_stream.py`
**职责**：统一解码 RTSP/MJPEG/图片/录像，提供帧获取、状态监控和 MJPEG 转发

**设计思路**：

```
VideoStreamService
├── 输入自适应
│   ├── rtsp://...   → OpenCV CAP_FFMPEG
│   ├── http://...   → OpenCV + MJPEG 解码
│   ├── *.jpg/*.png  → cv2.imread (静态图循环)
│   └── *.mp4/*.avi  → OpenCV VideoCapture (录像循环)
│
├── 帧缓存
│   ├── _latest_jpeg: bytes  (最新帧 JPEG 缓存)
│   └── _lock: threading.Lock (线程安全)
│
├── 状态监控 (VideoSnapshot)
│   ├── running, connected, frames_received, fps
│   ├── frame_width, frame_height
│   └── is_static_image, last_error
│
└── MJPEG 生成
    └── 原始/标注帧 → multipart/x-mixed-replace 流
```

**关键实现**：

- **类型自动识别**：`_open_source` 根据 URL 前缀判断——`rtsp://` 走 FFmpeg、`http://` 尝试 MJPEG stream 或 IP Webcam snapshot 模式、`.jpg/.png` 识别为静态图并以 2 FPS 循环发送同一帧；
- **断流恢复**：`_read_loop` 在流断开时自动重试连接（指数退避，最大间隔 10 秒）；
- **地址脱敏**：`_redact_url` 对日志和状态信息中的视频源地址进行脱敏，截去凭证和 IP 敏感段；
- **线程安全**：帧读写使用 `threading.Lock`，`latest_jpeg()` 返回缓存副本。

## 2. 视觉感知层

### 2.1 LocalModelService（本地模型推理）

**文件**：`backend/app/services/local_model.py`（71 个符号，最大单文件）
**职责**：YOLO 检测、ByteTrack 跟踪、车牌识别、白名单关联的完整推理流水线

**设计思路**：

```
LocalModelService.infer_frame(frame, camera_id, model_name, conf, imgsz)
│
├── 1. 视角判断 (_distant_view_crop)
│   └── 远景→裁剪道路区域→推理尺寸≥960
│
├── 2. YOLO 检测 (_detect_model)
│   ├── yolo11n (默认) / yolov11s_visdrone (远景) / yolov8n (fallback)
│   └── 输出: list[detection {bbox, class, confidence}]
│
├── 3. ByteTrack 跟踪 (_track_model)
│   ├── 持久轨迹、类别无关 NMS
│   └── 输出: track_id 关联的检测框
│
├── 4. 后处理过滤
│   ├── _filter_by_road_roi: 道路 ROI 过滤
│   ├── _filter_by_area: 最小面积过滤
│   ├── _recover_visual_ids: 视觉 ID 恢复
│   ├── _smooth_track_boxes: EMA 平滑
│   ├── _appearance_conf: 外观可信度过滤
│   └── _suppress_duplicates: 重叠框抑制
│
├── 5. 车牌识别 (detect_plates)
│   ├── 车辆裁剪 OCR: HyperLPR3 逐车识别
│   ├── OCR 预算控制: 每帧上限+轮转优先级
│   ├── 轨迹车牌记忆: 多帧稳定确认
│   ├── 全帧 OCR 补充: 周期全帧识别
│   └── 车牌到车辆关联/反推
│
├── 6. 白名单决策 (decide_plate)
│   ├── 稳定车牌→SQLite 白名单查询
│   ├── 决策缓存: 5秒防跳变
│   └── 输出: allow/deny + 原因 + 业务事件
│
├── 7. 速度估算 (_track_speed_cm_s)
│   ├── 框底边中心→透视变换→世界坐标
│   ├── 相邻观测位移÷时间差
│   ├── 死区过滤→中值平滑
│   └── 输出: speed_cm_s, speed_kmh
│
└── 8. 统计输出
    ├── 车辆数 (5帧中值)
    ├── 密度 (框面积÷ROI面积)
    └── AnalysisResult + 标注帧
```

**关键实现**：

- **模型懒加载**：`_load_model` 方法在首次调用时加载对应 YOLO 模型，支持按 `model_name` 切换（auto/visdrone/fallback）；
- **HyperLPR3 集成**：`_plate_catcher` 延迟初始化，双检测级别（低/高），实时流默认低级别以保速度；
- **流模式与单帧模式**：`infer_jpeg` 和 `infer_frame` 两个入口，实时流模式减少 OCR 开销并启用轨迹记忆，单帧模式做完整识别；
- **标注渲染**：`annotated_mjpeg_frames` 为 MJPEG 流注入标注框、车牌、白名单状态和速度信息。

### 2.2 RoadAnomalyService（道路异常识别）

**文件**：`backend/app/services/road_anomaly.py`（37 个符号）
**职责**：在道路区域检测不应出现的物体（异物、行人、破损）

**设计思路**：

```
RoadAnomalyService.analyze_frame(frame, camera_id, base_result)
│
├── 1. 状态初始化 (CameraState)
│   ├── baseline_gray: 稳定背景灰度图
│   ├── previous_gray: 上一帧灰度图
│   ├── frame_count: 帧计数器
│   ├── recent_legal_boxes: 近期合法目标框（防"空位"误报）
│   └── last_damage_at: 破损模型调用冷却
│
├── 2. ROI 生成 (_road_roi_mask)
│   ├── 固定摄像头: 预定义多边形 ROI
│   ├── live3 扩展: 针对远景增加 vivid_object 子 ROI
│   └── 自定义摄像头: 全画面或按宽高比估算
│
├── 3. 图像预处理 (_preprocess)
│   ├── 灰度转换
│   ├── CLAHE 对比度增强
│   └── 高斯平滑 (3×3)
│
├── 4. 实时流分支 (_frame_difference_candidates)
│   ├── 光流全局运动判断 (_global_motion_ok)
│   │   └── 中位位移>2.5px → 暂停检测
│   ├── 背景差分: |current - baseline|
│   ├── 自适应阈值: max(28, mean + 2.2×std)
│   ├── 变化比例检查: >8.5% → 跳过(光照/相机移动)
│   ├── 形态学: 开闭运算→去噪+连通
│   ├── 轮廓过滤: 面积/长宽/局部性
│   └── 合法目标扣除: YOLO框 + recent_legal_boxes
│
├── 5. 静态图分支 (_static_scene_candidates)
│   ├── 亮度通道: 自适应阈值提取高亮实体
│   ├── Lab 色度: 路面颜色→色差距离
│   ├── HSV 暖色: 橙黄小物体专项
│   ├── 纹理/形状: 填充率/圆度/凸包实心度
│   ├── 对比度/熵: 车道线/反光过滤
│   └── 白色占比: 标线特征
│
├── 6. 多帧确认 (_confirm_candidates)
│   ├── 实时流: 连续3帧 IoU≥0.30
│   ├── 静态图: 连续2帧 IoU≥0.35
│   └── 候选合并: IoU/覆盖率取高置信度
│
├── 7. 道路破损补充 (_road_damage_candidates)
│   ├── RDD YOLO12s 模型 (可选)
│   ├── Pothole YOLOv8 模型 (可选)
│   └── 冷却: 每摄像头每5秒最多调用1次
│
├── 8. 背景更新
│   └── 低权重(0.015) EMA更新，避免吸收异常物体
│
└── 9. 结果输出
    └── AnalysisResult (仅含异常类检测和事件)
```

**关键实现**：

- **CameraState 设计**：每个摄像头独立维护状态（背景模型、帧计数、合法框记忆），支持 `reset()` 重置；
- **道路破损模型**：作为可选补充（`include_damage_model` 参数），不用时不影响主流程；
- **任务隔离**：`_anomaly_only_result` 函数（main.py）确保异常模式输出不影响车辆监控指标；
- **JPEG API**：`analyze_jpeg` 方法接收 JPEG 字节流而非 numpy 数组，与 MJPEG 管道对齐。

### 2.3 RoadMaskService（道路语义分割）

**文件**：`backend/app/services/road_mask.py`（28 个符号）
**职责**：对输入帧进行道路区域语义分割，输出掩膜和道路示意底图

**设计思路**：

```
RoadMaskService
├── 模型: SegFormer-B0 (Cityscapes 预训练)
│   └── nvidia/segformer-b0-finetuned-cityscapes-1024-1024
│
├── 加载 (_load_model)
│   ├── AutoImageProcessor + AutoModelForSemanticSegmentation
│   ├── 道路类别提取: road_class_ids(model.config.id2label)
│   └── 设备: cuda:0 (优先) / cpu (降级)
│
├── 推理 (predict_mask)
│   ├── 输入: BGR frame → RGB → processor
│   ├── 输出: logits → argmax → 道路二值掩膜
│   └── 双线性插值恢复原图尺寸
│
├── 缓存
│   ├── 按摄像头 ID 缓存 (TTL 5秒)
│   └── 存储: (timestamp, mask_dict)
│
└── API 输出 (snapshot)
    ├── mask_data_url: PNG base64 Data URL
    ├── schematic: 道路示意底图
    └── dimensions: 原始宽高
```

**关键实现**：

- **按需下载**：`ensure_model_available()` 检测 `config.json` 存在性，缺失时自动 `snapshot_download`；
- **线程安全**：`threading.Lock` 保护模型加载和推理；
- **缓存机制**：避免每帧都做分割（语义分割耗时远大于检测），按摄像头 ID 和 TTL（5秒）缓存结果。

## 3. 交通分析层

### 3.1 RoadLogicService（道路逻辑与热力图）

**文件**：`backend/app/services/road_logic.py`（47 个符号）
**职责**：坐标映射、车道/路口归属、拥堵评分、禁停判断、热力图生成

**设计思路**：

```
RoadLogicService
├── 数据加载 (_load_json)
│   ├── road.json: 车道中心线定义
│   ├── camera.json: 摄像头标定点
│   ├── compact.json: 精简道路模型
│   └── intersection.json: 路口区域定义
│
├── 单应矩阵 (_build_homographies)
│   ├── _build_homographies_legacy: 4点标定→透视变换
│   └── _build_homographies: RANSAC增强→鲁棒单应
│
├── enrich(result: AnalysisResult)
│   ├── 坐标投影: 框底边中心→世界坐标
│   ├── 车道归属: 点到中心线最小距离
│   ├── 路口归属: 点在多边形内判断
│   ├── 活跃轨迹: 近期+几何可信的车辆
│   ├── 拥堵评分: 0.85×数量分 + 0.15×速度分
│   └── 禁停判断: 交通车道内持续停留
│
├── heatmap_snapshot(camera_id)
│   ├── 固定视角: 车道/路口拥堵热力状态
│   ├── 移动视角: 画面归一化热点
│   └── 输出: lane_stats[] + zone_stats[]
│
└── 热力点累积 (_heat_points)
    └── deque(maxlen=1200) 环形缓冲区
```

**关键实现**：

- **RANSAC 单应矩阵**：相比最小二乘，RANSAC 对异常标定点有更好的鲁棒性；
- **拥堵评分公式**：`score = 0.85 × countScore + 0.15 × speedScore`，以数量为主、速度为辅，避免单辆低速车染红整条车道；
- **禁停逻辑**：`_congestion_tracks` 记录车辆在交通车道内的持续停留时间，超过阈值（3秒）触发告警；
- **可靠框判断**：`_is_reliable_vehicle_box` 基于框尺寸、宽高比和位置判断是否为可信交通参与者；
- **摄像头质量记录**：`_camera_quality` 记录每个摄像头的标定点数量和单应矩阵状态，未标定摄像头走移动视角分支。

### 3.2 前端热力图渲染

**文件**：`frontend/src/heatmap.js`
**职责**：在 Canvas 上绘制道路热力图（画面叠加/道路示意图/关闭三种模式）

**关键实现**：

- **三种热力模式**（`heatmapMode`）：
  - `off`: 不绘制热力图；
  - `overlay`: 在原始画面上叠加半透明热点——固定视角用道路模型坐标，移动视角用归一化画面坐标；
  - `schematic`: 在独立道路示意图 Canvas 上绘制车道/路口拥堵色块；
- **热点累积与衰减**：检测到的车辆位置被加入热点缓冲区，随时间指数衰减（半衰期 3 秒）；
- **颜色映射**：绿色（畅通）→ 黄色（轻度拥堵）→ 橙色（中度拥堵）→ 红色（严重拥堵），按拥堵评分线性插值。

## 4. 数据服务层

### 4.1 AnalysisStore（检测历史与告警）

**文件**：`backend/app/services/analysis_store.py`（24 个符号）
**职责**：检测结果持久化、告警事件管理、证据存储和导出

**设计思路**：

```
AnalysisStore (SQLite: analysis_records, alert_incidents, evidence_frames)
│
├── 保存 (save)
│   ├── 序列化 detections/events/stats → JSON
│   ├── 写入 analysis_records
│   ├── 事件写入 alert_incidents (含 evidence JPEG)
│   └── 证据帧写入 evidence_frames (按 event_id)
│
├── 查询 (list_records, list_incidents)
│   ├── 按 camera_id、status 筛选
│   └── 分页 + 时间倒序
│
├── 处置 (update_incident)
│   ├── 状态流转: open → acknowledged → resolved
│   └── 记录处理人 + 备注
│
├── 导出 (export_csv, export_json)
│   └── 结构化 → CSV/JSON → Content-Disposition
│
└── 证据包 (get_incident_evidence)
    ├── 按 event_id 关联 records + incidents + frames
    └── ZIP: manifest.json + analysis-result.json + incident.json + frames + SHA-256
```

### 4.2 WhitelistStore（白名单管理）

**文件**：`backend/app/services/whitelist.py`（27 个符号）
**职责**：车牌白名单的持久化、查询和决策

**设计思路**：

- **车牌规范化**：`normalize_plate` 处理中文字符、大小写、空格和常见 OCR 错误；
- **精确+模糊匹配**：`match_plate` 先精确匹配，失败后对不含数字的字符做相似替换（如 O→0, I→1）；
- **内存缓存**：`_active_cache` 缓存启用状态的完整列表，60 秒过期，减少 SQLite 查询；
- **决策函数**：`decide_plate` 为独立纯函数，无副作用，可被前端和后端同时测试；
- **默认白名单**：`DEFAULT_WHITELIST_PLATES` 包含沙盘演示所需的初始车牌列表。

### 4.3 AuthStore（认证与权限）

**文件**：`backend/app/services/auth.py`（25 个符号）
**职责**：用户注册、登录认证、会话管理、密码安全和审计

**设计思路**：

- **密码哈希**：SHA-256 + 16 字节随机盐，`_hash_password(password, salt)` 返回 (salt, hash)；
- **会话管理**：`system_sessions` 表存储 token → user_id 映射，支持登出和过期清理；
- **验证码**：6 位字母数字，5 分钟有效，存储于内存 `_captchas` 字典；
- **用户管理**：`create_user, update_user, delete_user, list_users`，admin 可管理其他用户；
- **审计日志**：`audit_logs` 表记录所有管理操作（action, target, detail, username, timestamp）。

### 4.4 IntelligenceReportService（智能报告）

**文件**：`backend/app/services/intelligence_report.py`（22 个符号）
**职责**：DeepSeek API 配置管理和 AI 报告生成

**设计思路**：

- **配置隔离**：API key 写入后即不可读取（`_masked_key`），GET 接口只返回脱敏版本；
- **报告生成**：`generate(context, username)` 构建包含当前检测、拥堵、事件、天气和历史的综合 prompt，调用 DeepSeek chat API，归档结果至 `intelligence_reports` 表；
- **错误处理**：API 不可用时返回明确错误而不崩溃。

### 4.5 sqlite_utils（统一连接管理）

**文件**：`backend/app/services/sqlite_utils.py`（7 个符号）
**职责**：提供统一的 SQLite 连接类，确保资源正确释放

**设计思路**：

- **ClosingSQLiteConnection**：继承 `sqlite3.Connection`，重写 `__exit__` 方法——在提交/回滚后始终调用 `close()`；
- **open_sqlite**：工厂函数，默认使用 `ClosingSQLiteConnection` 和 `Row` 工厂；
- **解决 BUG-001**：此模块修复了 Windows 上 SQLite 连接未关闭导致临时数据库文件无法删除的问题。

## 5. 基础设施层

### 5.1 SystemMonitor（系统资源监控）

**文件**：`backend/app/services/system_monitor.py`（10 个符号）
**职责**：采集 CPU、内存、GPU、显存状态

**关键实现**：

- **psutil**：获取 CPU 利用率（`cpu_percent`）、内存（`virtual_memory`）；
- **pynvml**：获取 GPU 型号、利用率、显存占用、温度；
- **降级处理**：pynvml 不可用时 GPU 字段返回 `available: false`，不影响服务运行。

### 5.2 AdaptiveModelScheduler（自适应调度）

**文件**：`backend/app/services/adaptive_scheduler.py`
**职责**：根据系统资源和任务类型动态选择最优推理配置

**关键实现**（详见 [02-核心识别算法与流程设计.md](02-核心识别算法与流程设计.md) 第 5 节）：

- **五档策略**：quality / balanced / realtime / protect / anomaly
- **输入维度**：CPU%、内存%、GPU%、显存%、推理耗时、任务模式、静态/实时源
- **滞回机制**：8 秒冷却防止非 protect 策略频繁切换
- **资源缓存**：1.5 秒缓存避免 CPU 采集影响推理性能
- **决策历史**：保留最近 120 条策略变更，支持 `/api/model-scheduler` 查询

### 5.3 WeatherService（天气服务）

**文件**：`backend/app/services/weather.py`（14 个符号）
**职责**：获取北京天气信息，为智能报告和交通分析提供气象背景

**关键实现**：定期从公开 API 获取，本地缓存避免频繁请求。

### 5.4 AlgorithmClient（算法服务客户端）

**文件**：`backend/app/services/algorithm_client.py`（15 个符号）
**职责**：作为远程算法服务的抽象层，支持配置、健康检查和结果推送

**关键实现**：

- **双模式**：`enabled` + `base_url` 配置时走远程算法服务，否则使用本地模型；
- **结果缓存**：`latest_result` 存储最新分析结果，为前端轮询提供一致数据源；
- **事件缓存**：`events` 列表存储最近事件，供前端告警展示。

## 7. 前端架构

**文件**：`frontend/src/main.jsx`, `frontend/src/heatmap.js`, `frontend/src/cameraTypes.js`

**设计思路**：

### 7.1 单页大屏（main.jsx）

- 顶部导航：登录/用户信息、摄像头切换、任务模式切换
- 主区域：原始 MJPEG + 模型标注 MJPEG 双画面并排
- 侧面板：检测列表、交通统计、热力图控制、事件日志
- 底部：系统资源监控条（CPU/GPU/内存/推理耗时）
- 功能中心（模态页）：历史记录、告警处置、账号安全、白名单、摄像头管理、模型配置、自适应调度、用户管理、审计日志、智能报告
- 语音控制：Web Speech API 实现中文语音指令识别

### 7.2 热力图 Canvas（heatmap.js）

独立的 Canvas 渲染模块，支持三种热力图模式，通过 `cameraTypes.js` 获取当前摄像头支持列表。

### 7.3 摄像头类型（cameraTypes.js）

提取 `CAMERA_TYPE_OPTIONS` 常量，仅包含当前支持的摄像头类型（自定义、手机、USB、沙盘 RTSP），不再暴露已弃用选项（修复 BUG-002）。
