# STrans 系统设计说明书

## 1. 引言

### 1.1 项目概述

STrans（Smart Transportation Sensing System）是面向学院智慧交通沙盘的云边端协同视觉感知系统。系统接入多路网络摄像头、手机视频、本地图片或录像，在计算机端完成车辆检测与跟踪、车牌识别、白名单判断、道路异常分析、拥堵热力图和禁停告警，并通过 React Web 大屏呈现结果，形成"采集—识别—分析—处置—归档"的完整业务闭环。

### 1.2 文档目的

本文档描述 STrans 系统的整体架构设计、模块划分、接口约定、数据模型和关键设计决策，为结题答辩提供系统设计层面的支撑材料。

### 1.3 设计原则

1. **纵向闭环优先**：先保证从视频输入到前端展示的端到端链路可运行，再逐步增强算法能力；
2. **任务隔离**：车辆监控和道路异常作为两个独立任务模式，各自输出不互相污染；
3. **统一数据结构**：所有算法模块通过 `AnalysisResult` 统一输出，前端、持久化和算法服务使用同一数据契约；
4. **可降级运行**：远程算法不可用时使用本地模型；资源紧张时降尺寸/频率；GPU 不可用时使用 CPU；
5. **数据可追溯**：识别结果、事件、处置和证据通过统一的 `event_id` 关联，SHA-256 保证证据完整。

## 4. 数据模型设计

### 4.1 核心数据结构

#### AnalysisResult（分析结果）

```python
class AnalysisResult(BaseModel):
    frame_id: int = 0
    timestamp: str                       # ISO 8601
    camera_id: str
    model_id: str | None                 # 使用的模型标识
    inference_ms: float | None           # 推理耗时
    source_width: int = 0
    source_height: int = 0
    detections: list[DetectionBox]       # 检测目标列表
    traffic_stats: TrafficStats          # 交通统计
    events: list[TrafficEvent]           # 事件列表
    raw: dict | None                     # 扩展元数据
    error: str | None                    # 错误信息
```

#### DetectionBox（检测框）

```python
class DetectionBox(BaseModel):
    bbox: list[float]                    # [x1, y1, x2, y2]
    class_name: str
    confidence: float
    track_id: int | str | None           # ByteTrack ID
    plate: str | None                    # 车牌号
    whitelist_status: bool | None        # 白名单状态
    gate_action: "allow" | "deny" | None # 通行决策
    speed_cm_s: float | None             # 速度(厘米/秒)
    speed_kmh: float | None              # 速度(公里/小时)
    predicted: bool = False              # 是否为预测框
```

#### TrafficEvent（交通事件）

```python
class TrafficEvent(BaseModel):
    event_id: str                        # 唯一事件 ID
    type: str                            # road_obstacle, road_pedestrian,
                                          # road_damage, whitelist_pass,
                                          # whitelist_block, no_stop
    severity: "info" | "warning" | "critical"
    description: str
    camera_id: str
    bbox: list[float] | None
    timestamp: str | None
```

### 5.2 流式接口设计

- **MJPEG**：使用 `multipart/x-mixed-replace; boundary=frame`，每个 part 为 JPEG 帧；

- **模型流**：注入检测框、ID、车牌、白名单状态和异常标注，帧率受调度策略控制。

## 6. 安全设计

### 6.1 认证与授权

- **密码安全**：SHA-256 + 16 字节随机盐，管理员初始密码支持环境变量覆盖；
- **会话管理**：随机 token（32 字节 hex），支持主动登出；
- **验证码**：6 位字母数字，5 分钟有效期，防自动化注册/登录；
- **RBAC**：admin/user 两级角色，FastAPI `Depends` 注入权限检查；
- **自己的保护**：管理员不能禁用或删除自己的账号。

### 6.2 数据保护

- **视频地址脱敏**：日志和审计中的 `stream_url` 经 `_redact_url` 处理，不暴露完整 RTSP 凭据；
- **API Key 保护**：DeepSeek API Key 写入后不可读取，GET 只返回脱敏版本；
- **证据完整性**：SHA-256 哈希链保证证据帧未被替换；
- **CORS 限制**：仅允许 localhost 和局域网 IP 访问 API。

### 6.3 审计追踪

所有管理操作（用户管理、白名单变更、摄像头配置、告警处置、证据导出）均记录至 `audit_logs` 表，包含操作人、时间、动作、目标和详情。

## 9. 关键设计决策

### 9.1 为什么不使用端到端异常检测模型？

当前沙盘缺少大规模有标注的道路异常数据集。与其训练一个黑盒模型，不如采用可解释的多线索融合——模型负责解释正常道路使用者，几何和时序规则负责发现无法解释的变化。这种设计使得：
- 每一层过滤都有明确的物理含义；
- 阈值可以按场景独立调整；
- 错误案例可以追溯到具体是哪一层出了问题；
- 后续有足够数据时可替换其中某一层为学习模型。

### 9.2 为什么选择 SQLite 而非 MySQL/PostgreSQL？

- 项目定位为单机部署的沙盘监控系统；
- SQLite 零配置、零运维，适合演示和交付；
- 单文件数据库便于备份和迁移；
- 读写量在 SQLite 能力范围内（每 5 秒写一次分析记录）；
- 通过 `ClosingSQLiteConnection` 解决了连接管理问题。

### 9.3 为什么车辆监控和道路异常分为两个独立模式？

- 两种任务的业务目标不同：车辆监控关注交通流量，道路异常关注安全风险；
- 数据隔离：异常检测结果（道路异物、行人）不应进入车辆统计和热力图；
- 调度策略不同：道路异常需要保持较高分辨率以检测小目标；
- 前端展示不同：两种模式展示不同的指标面板和事件类型。

### 9.4 自适应调度的滞回机制为什么是 8 秒？

- 系统资源采样本身有波动（如瞬时 CPU 尖峰）；
- 模型切换本身有开销（加载权重、预热）；
- 8 秒滞回在"及时响应资源变化"和"避免频繁切换"之间取得平衡；
- protect 策略不受滞回限制——当资源接近上限时必须立即降级。
