# M08：数据闭环 — AnalysisStore + IntelligenceReportService

## 1. 模块定位

数据闭环由两个独立服务模块组成：

| 模块 | 文件 | 职责 |
|---|---|---|
| `AnalysisStore` | `analysis_store.py` (328行) | 检测历史持久化、告警事件管理、证据存储、CSV/JSON导出 |
| `IntelligenceReportService` | `intelligence_report.py` (199行) | DeepSeek API配置管理、AI报告生成与归档 |

## 2. AnalysisStore

### 2.1 数据库表结构

```sql
-- 检测记录主表
analysis_records (
    id, created_at, camera_id, frame_id, model_id,
    vehicle_count, current_count, count_in, count_out,
    density, avg_speed, congestion_level,
    detection_count, event_count, inference_ms,
    plates,                          -- 逗号分隔的车牌列表
    whitelist_pass_count,            -- 白名单通过数
    whitelist_block_count,           -- 白名单拦截数
    payload_json                     -- 完整 AnalysisResult JSON
)

-- 告警事件表
alert_incidents (
    event_id PRIMARY KEY,            -- UUID
    created_at, camera_id, event_type, severity, description,
    status DEFAULT 'pending',        -- pending|confirmed|resolved|false_positive
    handled_by, handled_at, note,    -- 处置信息
    analysis_record_id,              -- 外键 → analysis_records.id
    evidence_image_path              -- 证据帧文件路径
)
```

### 2.2 保存流程 (`save`)

```
1. 将 AnalysisResult 序列化为 payload_json
2. 提取车牌列表 (去重排序)、白名单统计
3. INSERT INTO analysis_records
4. 遍历 result.events:
   对每个事件:
     a. 保存证据帧: evidence/<event_id>.jpg
     b. INSERT OR IGNORE INTO alert_incidents
     c. UPDATE 补全 analysis_record_id + evidence_image_path
5. 返回 record_id
```

### 2.3 查询与导出

```python
list_records(limit, camera_id=None) → list[dict]
    # 按 id DESC 排序，支持按摄像头筛选
    # 返回聚合字段（不含完整 payload_json）

export_csv(limit=1000) → CSV字符串
export_json(limit=1000) → JSON字符串
    # 标准字段列表，不含大 JSON payload

count_records(camera_id=None) → int
delete_record(record_id) → bool
purge_records(before=None) → 删除数
```

### 2.4 告警处置

```python
list_incidents(limit=100, status=None)
    # 支持按状态筛选: pending/confirmed/resolved/false_positive
    # 返回字段含 has_snapshot (是否有证据帧)

update_incident(event_id, status, handled_by, note)
    # 状态流转: 四条法律状态
    # 记录处置人、时间和备注
    # raise ValueError on 无效状态/不存在的 event_id
```

### 2.5 证据包 (`get_incident_evidence`)

```
按 event_id 关联:
  alert_incidents → analysis_record_id → analysis_records
  → payload_json (完整结构化结果)
  → evidence_image_path → 原始图像 bytes
返回 {
    incident: {...},       # 告警记录
    analysis: {...},       # 关联的分析结果
    image: bytes | None,   # 原始帧
    image_sha256: str,     # 图像哈希
}
```

证据包下载 (main.py `download_incident_evidence`):
```
ZIP内容:
  manifest.json              # 包清单 + 哈希
  analysis-result.json       # 检测结果
  incident.json              # 告警元数据
  evidence-frame-original.jpg
  evidence-frame-annotated.jpg  # 标注版
  README.txt
```

### 2.6 向后兼容

`_init_db` 使用 `ALTER TABLE ... ADD COLUMN` 的 `try/except` 模式，支持从旧版数据库平滑升级。同时运行一次历史事件迁移（最近 200 条记录的 `payload_json` 中的事件写入 `alert_incidents`）。

## 3. IntelligenceReportService

### 3.1 配置管理

```sql
intelligence_config (
    id INTEGER PRIMARY KEY CHECK (id = 1),  -- 单例行
    api_base TEXT,       -- 默认 https://api.deepseek.com/v1
    model TEXT,          -- 默认 deepseek-chat
    api_key TEXT,        -- API密钥(写入后不可读回)
    updated_at TEXT
)
```

**安全设计**：
- `get_config()` 返回 `api_key_masked`: `"*"*(len-4) + last_4_chars`
- `update_config()` 当 `api_key` 为 `None` 或空时保持旧值
- `api_base` 必须 `http://` 或 `https://` 开头

### 3.2 报告生成 (`generate`)

```
1. 读取配置 → 若无API Key → raise ValueError
2. _build_prompt(context):
    包含指令(5个固定章节) + JSON快照
3. POST https://api.deepseek.com/v1/chat/completions
   model: deepseek-chat
   temperature: 0.25
   max_tokens: 1300
   system: "你是交通监测系统分析助手。只能依据提供的数据总结，不得编造..."
   messages: [{"role":"user","content":prompt}]
   timeout: 35s
4. 提取 choices[0].message.content
5. INSERT INTO intelligence_reports
6. 返回 {id, created_at, title, content, model, input_summary}
```

### 3.3 Prompt 模板

```
报告固定包含：
1. 总体态势（1段）
2. 检测摘要（车辆、车牌、拥堵、速度、推理耗时）
3. 事件与风险（只列有证据的事件）
4. 处置建议（2-4条，可执行）
5. 数据局限说明（如静态画面、模型误检、车牌不稳定等）

不要重复JSON；不要将"未检测到"写成"不存在"；数据不足时明确说明。

监测快照：[JSON context]
```

**设计意图**：`temperature=0.25` 保证输出的一致性；`max_tokens=1300` 控制报告长度；system prompt 强调"只能依据数据"防止 AI 编造。

## 4. API 接口汇总

### 历史记录

| 端点 | 方法 | 权限 | 说明 |
|---|---|---|---|
| `/api/history` | GET | user | 查询 + 分页 + 筛选 |
| `/api/history/export` | GET | user | CSV/JSON 导出 |
| `/api/history/{id}` | DELETE | admin | 删除单条 |
| `/api/history` | DELETE | admin | 批量清理 |

### 告警事件

| 端点 | 方法 | 权限 | 说明 |
|---|---|---|---|
| `/api/incidents` | GET | user | 告警列表 + 状态筛选 |
| `/api/incidents/{event_id}` | PUT | user | 处置告警(状态+备注) |
| `/api/incidents/{event_id}/evidence` | GET | user | 证据包下载(ZIP) |

### 智能报告

| 端点 | 方法 | 权限 | 说明 |
|---|---|---|---|
| `/api/intelligence/config` | GET | admin | 查看配置(密钥脱敏) |
| `/api/intelligence/config` | PUT | admin | 更新配置 |
| `/api/intelligence/reports` | GET | user | 报告列表 |
| `/api/intelligence/reports` | POST | admin | 生成报告 |
| `/api/intelligence/reports/{id}` | DELETE | admin | 删除报告 |

## 5. 数据闭环流

```
实时帧推理
    ↓
handle_traffic_result() / anomaly_frames()
    ↓
每5秒: analysis_store.save(result, source_jpeg)
    ├── analysis_records ← 结构化字段 + payload_json
    ├── alert_incidents ← 事件(含 evidence_image_path)
    └── evidence/xxx.jpg ← 原始帧
    ↓
用户操作:
    查看历史 → GET /api/history
    导出数据 → GET /api/history/export?format=csv
    处置告警 → PUT /api/incidents/{id} {status, note}
    下载证据 → GET /api/incidents/{id}/evidence → ZIP
    生成报告 → POST /api/intelligence/reports → DeepSeek → 归档
    审计追踪 → 所有管理操作 → audit_logs
```
