# M04：道路逻辑与热力图 — RoadLogicService

## 1. 模块定位

`RoadLogicService` (`backend/app/services/road_logic.py`，47 符号，595 行) 负责将图像空间的检测结果映射到统一道路坐标系，并从中推导车道/路口归属、拥堵状态、禁停告警和热力图数据。它是"视觉感知"与"交通分析"之间的桥梁。

## 2. 架构总览

```
RoadLogicService
├── 数据层 (4个JSON文件)
│   ├── road-lane.v1.json:      车道中心线、节点、路段拓扑
│   ├── camera-mapping.v1.json: 每路摄像头标定点(image→world)
│   ├── road-model.compact.json: 精简模型(摄像头绑定streamPresetId)
│   └── intersection-zones.v1.json: 路口多边形
│
├── 计算层
│   ├── _homographies: dict[camera_id → 3×3矩阵]
│   │   ├── ≥4点: cv2.findHomography + RANSAC
│   │   └── 2-3点: cv2.estimateAffinePartial2D → 相似变换
│   └── _camera_quality: dict[camera_id → 标定质量元数据]
│
├── 状态层 (跨帧)
│   ├── _heat_points: deque(1200)      # 热力点环形缓冲
│   ├── _congestion_tracks: dict        # 活跃车辆(3s TTL)
│   └── _stop_states: dict             # 禁停检测状态
│
└── 核心入口
    ├── enrich(result: AnalysisResult) → AnalysisResult
    └── heatmap_snapshot(camera_id?) → dict
```

## 3. 核心算法

### 3.1 标定矩阵构建 (`_build_homographies`)

```
步骤:
1. 遍历 camera-mapping 中的每个 camera:
    取 ≥2 个标定点
    匹配 compact model 中的 streamPresetId
2. 同一 stream 可能有多个 camera 记录 → 取标定点多的
3. ≥4个标定点:
   cv2.findHomography(points, world, cv2.RANSAC, 12.0)
   方法: "homography_ransac"
4. 2-3个标定点:
   cv2.estimateAffinePartial2D(points, world, cv2.LMEDS)
   → 扩展为 3×3 矩阵 [R|t; 0,0,1]
   方法: "similarity_2point" / "affine_partial"
5. 记录 quality 元数据:
   - ready: 是否成功拟合
   - inlier_count: RANSAC 内点数
   - method: 使用的拟合方法
   - point_count: 标定点数量
   - ambiguous: 是否有多条cam记录竞争同一stream
```

**关键设计——2点标定的相似变换降级**：
不是所有摄像头都有足够的标定点。当只有2-3个点时，使用 `estimateAffinePartial2D`（相似变换：旋转+统一缩放+平移）作为降级方案。相比直接放弃，它能提供可用的方向性映射。

### 3.2 坐标投影 (`enrich`)

```
对每个车辆检测框:
1. 取框底边中心作为路面接触点
   contact_x = (x1+x2)/2 * (1920/source_w)
   contact_y = y2 * (1080/source_h)
   注意: 归一化到标定分辨率 1920×1080
2. 透视变换: world_point = H × [contact_x, contact_y, 1]
3. 世界坐标检查: inside_world(point)
4. 找最近车道: _nearest_lane(point)
   → 点到每条车道中心线(renderPath)的距离
   → 距离 ≤ lane_width/2 + 6cm → on_lane
5. 路口判断: _junction_for_point(point)
   → 射线法判断点是否在路口多边形内
6. 综合: on_modeled_road = on_lane or in_junction
```

### 3.3 拥堵评分 (`heatmap_snapshot`)

```
对每个活跃轨迹:
1. 按 lane_id / junction_id 分组
2. 每组聚合: track_id 集合 + speed 列表
3. 评分公式:
   countScore = min(1.0, track_count / 4.0)      # 4辆车→满分
   speedScore = max(0, 1.0 - min(avg_speed/18, 1)) # 18cm/s→满分
   score = 0.85 × countScore + 0.15 × speedScore
4. 等级:
   score < 0.20 → free       (畅通)
   score < 0.42 → smooth     (基本畅通)
   score < 0.62 → slow       (缓行)
   score < 0.82 → congested  (拥堵)
   score ≥ 0.82 → severe     (严重拥堵)
```

**设计意图——数量主导**：
- 0.85/0.15 的比例确保单辆低速车不会把整条车道染红
- 拥堵需要**多辆已确认轨迹**(≥ CONGESTION_CONFIRM_FRAMES=2帧)，而不是同一辆车的重复检测

### 3.4 可靠车辆框判断 (`_is_reliable_vehicle_box`)

```
可靠条件:
  宽 ≥ 16px 且 高 ≥ 16px
  面积 ≥ 700px²
  宽高比 ∈ [0.22, 3.6]
```

**设计意图**：排除被误检为 car 的细小路面标记（箭头/标线），这些"假车"不应影响拥堵和热力图。

### 3.5 禁停检测 (`_update_stop_state`)

```
对每个在交通车道内(非路口)的轨迹:
1. 记录初始位置
2. 每帧检查: 位移 < 6cm → 持续计时
3. 位移 > 6cm → 重置计时和位置
4. dwell ≥ 30秒 → 触发 "illegal_stop" 事件
5. 每条轨迹只触发一次(alerted标志)
6. 轨迹消失3秒 → 清理
```

## 6. 关键设计决策

### 6.1 接触点选择

```
框中心点 vs 框底边中心:
  透视视角下，车辆顶部延伸出车道边界
  → 底边82%位置更贴近路面
  → 框底被画面裁剪时降级为框中心
```

### 6.2 热力图不是热力图

```
传统理解: 累积所有历史检测点 → 越来越红的区域
实际实现: 只保留 8 秒内的活跃轨迹
  → 描述"此刻的交通状态"，不是"历史总量"
  → HEAT_TTL_SECONDS = 8.0, deque_maxlen = 1200
```

### 6.3 速度归一化

```
speedScore = 1.0 - min(speed / 18.0, 1.0)
  → 沙盘比例: 18 cm/s ≈ 真实 58 km/h (1:87比例)
  → 超过18cm/s → speedScore=0 (不影响拥堵评分)
```
