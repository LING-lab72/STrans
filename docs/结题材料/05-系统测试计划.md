## 2. 测试目标

1. 验证系统能够完成视频接入、视觉识别、道路分析、业务处置和数据归档的主流程；
2. 核验车辆、车牌、道路异常、道路掩膜和拥堵热力图等核心算法模块的可重复行为；
3. 验证普通用户与管理员权限边界、敏感信息保护和审计记录；
4. 验证历史、白名单、告警、摄像头、模型配置和智能报告的数据持久化；
5. 通过真实浏览器遍历发现页面、接口、控制台、网络和交互问题；
6. 形成可用于结题报告和答辩的测试证据、缺陷清单和遗留风险说明。

### 3.2 不纳入当前正式验证

- 未在当前主链路实现的独立碰撞识别、信号灯识别和完整跨线流量算法；
- 物理闸机、外部道路设备等未接入硬件的动作执行；
- 外部智能报告服务自身的模型质量；只验证本系统请求、错误处理和归档；
- 公网、校园网或第三方服务不可控故障的长期稳定性。

### 4.2 覆盖完整性结论

当前测试集能够证明若干局部工具函数，但尚不能证明系统主业务闭环。主要缺口为：

- 核心算法 `LocalModelService`、`RoadAnomalyService`、`RoadLogicService` 缺少行为级单元测试；
- 白名单、认证、分析历史、告警证据、摄像头配置等 SQLite 服务缺少隔离数据库测试；
- 自适应模型调度和远程算法客户端缺少边界条件测试；
- FastAPI 主要接口没有集成测试，权限依赖和错误码未被自动验证；
- React 主页面没有组件测试，当前只能通过浏览器系统遍历验证；
- 没有覆盖率工具和覆盖率阈值；
- 没有面向真实识别数据的可重复算法回归集；
- 没有性能基线、长时间运行和恢复性测试。

因此，现有测试完整性评级为：**局部基础测试可用，系统级覆盖不足，高风险核心路径需要补测**。

## 5. 测试策略

### 5.1 测试分层

| 层级 | 目标比例 | 方法 | 运行频率 |
|---|---:|---|---|
| 单元测试 | 约 70% | Python `unittest`、Node `node:test`，测试纯函数和服务行为 | 每次修改后 |
| 集成测试 | 约 20% | FastAPI TestClient、临时 SQLite、真实服务组合、外部依赖替身 | 每个功能切片后 |
| 系统/E2E | 约 10% | 本地前后端 + Playwright，遍历关键用户流程 | 系统里程碑和交付前 |
| 识别评测 | 独立统计 | F 盘样本、固定模型配置、结果与人工真值比对 | 模型或规则变更后 |

### 5.2 测试原则

- 使用临时目录和临时 SQLite，禁止污染交付数据库；
- 优先断言输入输出和持久化状态，不绑定内部调用顺序；
- 模型、网络、摄像头和智能报告等慢或不稳定边界使用可控替身；
- 核心几何、阈值和状态机使用确定性合成数据；
- 新发现缺陷先添加失败复现测试，再修复实现；
- 每个测试独立运行，不能依赖执行顺序或前一用例残留状态；
- 正式系统遍历保留截图、控制台错误和网络失败证据。

## 8. 增量补测计划

### 阶段 A：测试基础与纯逻辑

1. 使后端现有 `unittest` 可一条命令发现并运行；

### 阶段 B：SQLite 服务与 API

1. 使用临时数据库补认证、白名单、历史和告警持久化测试；
2. 补 CameraHub 配置持久化测试；

验收：测试不写入 `backend/data/traffic_analysis.db`，关键 API 的成功、未认证、越权和无资源分支均被验证。

### 阶段 C：页面与系统遍历

1. 启动专用测试后端和前端；
2. 使用 Playwright 遍历登录、实时监控和功能中心；
3. 收集控制台、网络、DOM 和截图证据；
4. 对发现的缺陷先补复现测试，再决定是否修复。

验收：P0/P1 用户流程都有执行记录和证据文件。

### 阶段 D：真实识别样本

1. 只读盘点 F 盘样本；

3. 固定模型、阈值和尺寸批量执行；

验收：报告能区分正确、误报、漏报和环境阻塞，所有统计可由原始结果复算。

## 11. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|---|---|---|
| 本地环境缺少测试依赖 | 后端测试无法启动 | 优先使用标准库 `unittest`；必要依赖写入测试说明 |
| 模型大、初始化慢 | 自动化测试超时 | 纯逻辑测试不加载模型，模型测试单独分组 |
| 视频源不稳定 | 系统用例波动 | 使用本地固定图片/录像作为主基线，实时源作为补充 |
| 全局服务在导入时创建真实数据库 | 污染交付数据 | 集成测试前隔离工作目录或注入临时服务实例 |
| 规则阈值依赖场景 | 合成测试通过但真实场景失败 | 合成单元测试和 F 盘真实样本评测同时保留 |
| 浏览器语音能力受环境限制 | 语音用例不稳定 | 验证入口和权限提示，麦克风识别作为环境相关项 |
| 外部道路建模目录未随仓库交付 | 功能入口可见但换机后页面缺失 | 将主页面最小运行集纳入 `frontend/public`，以资产完整性、模型逻辑和浏览器加载三层门禁验证 |
| Python 环境依赖不一致 | 测试收集失败或 CUDA 不可用 | 主项目固定使用 CUDA 可用的 Python 3.14；建模分析脚本使用具备 orjson/Pillow/Shapely 的 Anaconda 环境，并在报告中分别记录 |

### 13.3 完备性判定方法

“全量测试通过”只表示已列用例全部执行成功，不等同于所有需求均已获得自动化覆盖。完备性分三层判断：

- **功能完备性**：视频接入、识别、道路分析、管理闭环和道路建模均有可运行入口；
- **证据完备性**：每个主要模块至少有代码结构、自动化测试、浏览器或真实数据证据之一；
- **覆盖完备性**：仍需补齐 API/RBAC 状态码矩阵、`RoadAnomalyService` 直接回归、`VideoStreamService` 生命周期、断流恢复和长稳压力测试。

因此结题口径为“主要功能链路完整且可验证，已知精度和长稳风险透明”，不表述为“所有需求和算法准确率均达到 100%”。
